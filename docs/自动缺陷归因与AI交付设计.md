## 1. 设计目标

AutoTest 的价值不应止步于“测出错了”，而应进一步回答:

- 出了什么问题
- 在哪一步出现
- 证据是什么
- 可能根因是什么
- 应该优先修哪里
- 修完后如何回归

因此，本文档要解决两件事:

1. 如何把多维异常信号收敛为缺陷对象
2. 如何把缺陷对象交付成开发者和 AI 助手可直接消费的结果包

---

## 2. 为什么传统报告不够

传统自动化报告常见问题:

- 只给出 pass/fail
- 截图、日志、网络数据分散
- 缺乏跨维度因果链
- 开发者需要重新复现
- AI 助手难以直接理解

AutoTest 的目标不是“更好看的报告”，而是“更可修复的结果”。

---

## 3. 总体设计主线

缺陷归因与 AI 交付的完整链路建议定义为:

```text
步骤执行结果
  -> 多维信号收集
  -> 异常检测
  -> 信号聚合
  -> 缺陷归因
  -> 修复上下文生成
  -> 多视图交付
  -> 回归入口生成
```

---

## 4. 缺陷归因流程

### 4.1 阶段 1: 原始信号收集

输入来源建议包括:

- 步骤动作结果
- 页面截图
- OCR 文本
- DOM / 页面状态
- 控制台日志
- 网络请求
- 业务断言结果
- 任务上下文

### 4.2 阶段 2: 异常信号识别

系统需要先回答:

- 哪些信号异常
- 哪些异常可靠
- 哪些异常可能只是噪声

建议区分:

- 确定性异常
- 高概率异常
- 可疑异常
- 噪声信号

### 4.3 阶段 3: 异常聚合

同一问题常常在多个维度同时出现。

例如:

- 登录失败
- 登录接口 500
- 控制台报错
- 页面停留不动

这些不应被拆成完全独立的 4 个问题，而应先判断它们是否属于同一缺陷簇。

### 4.4 阶段 4: 根因推断

在聚合后，系统再尝试回答:

- 最可能的触发点是什么
- 是前端问题还是后端问题
- 是环境问题还是业务问题
- 哪个信号更接近“根”

### 4.5 阶段 5: 修复交付

归因完成后，系统应生成:

- 缺陷标题
- 严重级别
- 影响范围
- 复现路径
- 证据链
- 根因推断
- 修复建议
- 回归建议
- AI 助手结构化输入

## 5. 缺陷对象模型

建议每个缺陷都被定义为一个一等对象，而不是报告中的附属条目。

### 5.1 Defect 基本信息

| 字段 | 说明 |
|---|---|
| `defect_id` | 缺陷唯一标识，建议前缀 `def_` |
| `task_id` | 所属任务 |
| `run_id` | 所属执行 |
| `title` | 缺陷标题 |
| `severity` | 严重程度 |
| `status` | 状态 |
| `category` | 缺陷类型 |
| `confidence` | 归因置信度 |
| `created_at` | 创建时间 |

### 5.2 Defect 语义信息

| 字段 | 说明 |
|---|---|
| `symptom` | 表现现象 |
| `impact_scope` | 影响范围 |
| `business_context` | 业务上下文 |
| `likely_root_cause` | 最可能根因 |
| `root_cause_candidates` | 根因候选集合 |
| `repair_suggestion` | 修复建议 |
| `regression_suggestion` | 回归建议 |

### 5.3 Defect 证据入口

| 字段 | 说明 |
|---|---|
| `step_refs` | 关联步骤 |
| `screenshots` | 截图证据 |
| `console_refs` | 控制台证据 |
| `network_refs` | 网络证据 |
| `page_state_refs` | 页面状态证据 |
| `evidence_chain` | 证据链对象 |

## 7. 严重级别定义

建议统一为四级:

- `critical`
- `high`
- `medium`
- `low`

### 7.1 critical

适用于:

- 核心流程完全阻断
- 页面白屏、崩溃
- 登录不可用
- 主功能不可达

### 7.2 high

适用于:

- 核心功能明显错误
- 关键接口失败但页面未完全崩溃
- 关键权限错误

### 7.3 medium

适用于:

- 非核心功能异常
- 页面局部错误
- 文案和交互问题

### 7.4 low

适用于:

- 次要视觉问题
- 轻微提示异常
- 对流程影响较低的 warning

## 8. 证据链模型

缺陷可信度的核心不在于语言描述，而在于证据链是否完整。

### 8.1 EvidenceChain 的作用

`EvidenceChain` 用于把分散证据串成可解释的链路。

它应当回答:

- 什么动作触发了问题
- 当时页面是什么状态
- 控制台发生了什么
- 网络发生了什么
- 业务结果为什么被判定异常

### 8.2 证据链组成

建议包括:

- 触发步骤
- 触发前截图
- 触发后截图
- OCR 摘要
- 页面状态摘要
- 控制台异常摘要
- 网络请求摘要
- 断言偏差说明
- 因果链说明

### 8.3 证据链示例

```text
步骤: 点击“登录”
  -> 截图显示页面未跳转
  -> OCR 未发现欢迎文案
  -> 控制台出现 TypeError
  -> /api/login 返回 500
  -> 业务断言“进入首页”失败
结论:
  登录接口失败导致前端异常，登录链路中断
```

## 9. 根因推断模型

### 9.1 根因不是“猜测一句话”

根因推断应是一个有结构的对象，而不是简单自然语言。

建议至少包括:

- `primary_hypothesis`
- `supporting_signals`
- `conflicting_signals`
- `confidence`
- `next_checkpoints`

### 9.2 根因推断逻辑

建议遵循以下优先级:

1. 优先识别确定性外部错误
2. 再识别强关联信号
3. 最后生成候选根因

例如:

- 如果接口 500 且后续页面异常，优先判为后端触发
- 如果接口正常但控制台报错且页面渲染失败，优先判为前端触发
- 如果所有业务都失败但环境登录失效，优先判为环境阻塞

### 9.3 防止过度归因

当证据不足时，不应输出确定性结论。

建议支持:

- “最可能根因”
- “候选根因 1/2/3”
- “需要进一步确认的点”

## 10. 修复上下文模型

建议定义独立对象:

**`RepairContext`**

该对象专门服务开发者与 AI 助手。

### 10.1 RepairContext 目标

让消费方在不回看全量执行日志的前提下，也能快速理解问题并尝试修复。

### 10.2 建议内容

- 缺陷标题
- 任务背景
- 业务目标
- 复现步骤
- 实际表现
- 期望结果
- 页面截图
- OCR 文本摘要
- 控制台错误摘要
- 网络异常摘要
- 根因候选
- 修复建议
- 回归建议

### 10.3 设计原则

- 对开发者: 强调定位效率
- 对 AI 助手: 强调结构化、上下文充足、歧义少

## 11. AI 交付模型

AutoTest 的交付对象不应只有“页面报告”，还应包括 AI 可直接消费的数据包。

### 11.1 三类交付对象

#### 面向测试人员

- 任务结果摘要
- 缺陷清单
- 步骤与截图

#### 面向开发者

- 缺陷详情
- 证据链
- 根因推断
- 修复建议

#### 面向 AI 助手

- 结构化 JSON
- MCP 对象
- RepairContext

### 11.2 AI 助手最小可用输入

建议至少包括:

- `task_summary`
- `defect_summary`
- `reproduction_steps`
- `expected_result`
- `actual_result`
- `console_errors`
- `network_failures`
- `page_state`
- `root_cause_candidates`
- `repair_suggestion`

---

## 12. MCP / JSON 交付建议

### 12.1 JSON 结果包结构建议

```json
{
  "task": {},
  "run": {},
  "defects": [],
  "delivery": {
    "developer_view": {},
    "ai_context": {}
  }
}
```

### 12.2 MCP 建议能力

建议优先暴露:

- `get_task(task_id)`
- `get_task_delivery(task_id)`
- `list_task_defects(task_id)`
- `get_defect(defect_id)`
- `get_repair_context(defect_id)`

### 12.3 设计原则

- MCP 返回不应是页面片段
- 应直接返回结构化对象
- 应避免要求 AI 助手自己拼接不同接口结果

## 13. 回归包设计

AutoTest 的结果交付不应停留在“发现缺陷”，还应支持下一轮最小回归。

### 13.1 RegressionPackage

建议为每个缺陷生成最小回归包。

内容建议包括:

- 缺陷对应链路
- 最小复现步骤
- 必测断言
- 修复后重点验证点

### 13.2 用途

- 修复后快速复测
- CI 中只跑高风险回归
- AI 助手修复后自动触发局部验证

## 15. 风险控制

### 15.1 误报风险

来源:

- OCR 误识别
- 临时网络波动
- 三方脚本干扰

建议:

- 缺陷对象中保留置信度
- 降低单一弱信号的权重

### 15.2 误归因风险

来源:

- 同一时刻多个异常并发出现
- 真正根因不在当前采集范围内

建议:

- 允许多候选根因
- 不足证据时避免强结论

### 15.3 交付不完整风险

来源:

- 报告偏展示，缺少结构化上下文

建议:

- 先产出结构化对象，再做页面渲染

---

## 16. 与任务模型的关系

`Defect` 和 `RepairContext` 应挂载到 `TestTask` 上，但职责应分离:

- `TestTask`: 管理任务级生命周期与交付入口
- `Defect`: 管理问题对象
- `RepairContext`: 管理开发与 AI 修复上下文

这种设计可以保证:

- 任务视角清楚
- 缺陷视角独立
- 修复交付单独优化
