## 1. 设计目标

本文档解决的不是某个局部模块设计问题，而是定义 AutoTest 的唯一主线:

**用户输入目标后，系统如何自动完成测试，并稳定交付一份可直接用于修复问题的结果。**

为避免产品能力继续分散，本文明确以下设计目标:

1. 定义统一的全自动 AI 测试生命周期
2. 统一快速模式与文档驱动模式的闭环骨架
3. 统一每一阶段的输入、系统动作、产物与降级策略
4. 统一什么叫“完成一次全自动测试”
5. 统一最终交付给测试人员、开发者与 AI 助手的结果模型

---

## 2. 闭环的本质定义

### 2.1 什么叫“全自动 AI 测试闭环”

AutoTest 的闭环不是指“执行完若干测试步骤”，而是指以下链路完整跑通:

```text
输入目标
  -> 自动准备
  -> 自动理解
  -> 自动规划
  -> 自动执行
  -> 自动校验
  -> 自动归因
  -> 自动交付
```

只有当系统完成了以上全过程，并产出可消费结果时，才应被判定为一次完整闭环。

### 2.2 什么不算闭环

以下情况都不应被视为闭环完成:

- 只创建了项目，但没有开始测试
- 只跑了浏览器操作，没有形成结构化判断
- 只输出了 pass/fail，没有给出证据与归因
- 只发现错误，没有形成开发可消费的修复上下文
- 因环境问题中断，但没有说明阻塞原因与下一步建议

### 2.3 闭环产出标准

一次完整闭环至少应输出:

- 测试任务摘要
- 自动化执行路径
- 步骤级记录
- 多维校验结果
- 缺陷列表
- 缺陷证据链
- 根因推断
- 修复建议
- 供 AI 助手直接消费的结构化结果

## 3. 闭环设计原则

### 3.1 目标优先，不以项目管理为中心

产品视角应从“项目/文档/场景/执行”切换为“自动测试任务”。

系统内部仍可以保留项目、知识库、场景等对象，但用户看到的主线应始终是:

```text
我要测试什么 -> 系统正在做什么 -> 结果是什么 -> 如何修复
```

### 3.2 自动优先，人工介入作为降级而非默认

闭环中的每个阶段都要优先设计自动路径，其次才是人工补充路径。

设计原则:

- 能自动识别的，不要求人工填写
- 能自动恢复的，不仅提示，还要自动处理
- 需要人工介入的，必须清楚说明介入原因与影响范围

### 3.3 结构化优先，展示层其次

所有阶段产物都应先形成结构化对象，再派生:

- UI 页面
- 执行日志
- HTML 报告
- MCP 输出

不允许只存在“页面展示结果”，而没有可复用的结构化对象。

### 3.4 可降级但不可失控

AutoTest 必须支持降级运行，但降级必须可解释、可追踪、可度量。

例如:

- 无文档时降级为页面发现模式
- 无 AI key 时降级为规则引擎 + 基础校验
- 某类采集失败时，仍保留其他维度校验

### 3.5 结果必须服务修复

闭环最终不是为了证明“测试做过了”，而是为了帮助修复。

因此最终输出必须优先服务:

- 开发者理解问题
- AI 助手自动消费数据
- 下一轮回归验证

## 4. 总体闭环模型

### 4.1 八阶段模型

建议 AutoTest 使用固定的八阶段闭环:

1. 测试入口
2. 环境预检
3. 测试对象理解
4. 测试蓝图生成
5. 执行编排
6. 多维校验
7. 缺陷归因
8. 结果交付

### 4.2 闭环全景图

```text
用户输入目标
  |
  v
[阶段1 测试入口]
  |
  v
[阶段2 环境预检]
  |---- 若阻塞 -> 输出阻塞原因与补充建议
  |
  v
[阶段3 测试对象理解]
  |---- 无文档 -> 页面发现路径
  |---- 有文档 -> 文档理解路径
  |
  v
[阶段4 测试蓝图生成]
  |---- 低置信度 -> 生成最小可执行蓝图
  |
  v
[阶段5 执行编排]
  |---- 单步失败 -> 重试 / 恢复 / 局部跳过
  |
  v
[阶段6 多维校验]
  |---- 某维缺失 -> 其余维继续运行并标注缺口
  |
  v
[阶段7 缺陷归因]
  |---- 无法确认根因 -> 输出候选根因集合
  |
  v
[阶段8 结果交付]
  -> 给测试人员的视图
  -> 给开发者的视图
  -> 给 AI 助手的结构化输出
```

## 5. 两种运行模式如何统一进同一闭环

### 5.1 零文档快速模式

适用场景:

- 只有 URL
- 只想快速冒烟
- 没有 PRD / API 文档
- 希望分钟级得到结果

该模式下的核心路径:

```text
URL
  -> 页面发现
  -> 通用测试蓝图
  -> 自动执行
  -> 基础多维校验
  -> 结构化问题输出
```

### 5.2 文档驱动完整模式

适用场景:

- 有 PRD / UI 规范 / API 文档
- 希望验证业务流程
- 希望更高覆盖、更强归因
- 希望交付更强的 AI 修复上下文

该模式下的核心路径:

```text
URL + 文档
  -> 文档理解 + 页面理解
  -> 业务测试蓝图
  -> 自动执行
  -> 完整多维校验
  -> 因果链级缺陷输出
```

### 5.3 统一骨架，不同深度

两种模式的差别，不应是是否走同一流程，而应是各阶段深度不同:

| 阶段 | 零文档快速模式 | 文档驱动完整模式 |
|---|---|---|
| 测试入口 | URL 为主 | URL + 多文档 |
| 理解阶段 | 页面理解为主 | 文档 + 页面联合理解 |
| 蓝图阶段 | 通用冒烟蓝图 | 业务链路蓝图 |
| 校验阶段 | 基础四维 | 深度四维 + 业务规则 |
| 归因阶段 | 单点异常为主 | 因果链归因为主 |
| 交付阶段 | 快速定位 | 修复上下文更完整 |

## 6. 阶段设计详解

以下每一阶段都统一描述:

- 目标
- 输入
- 系统动作
- 输出
- 完成标准
- 失败与降级策略

### 6.1 阶段 1: 测试入口

#### 目标

把用户意图转换为系统可执行的测试任务。

#### 输入

- 被测 URL / App 入口
- 平台类型
- 可选文档
- 可选目标范围
- 可选账号信息

#### 系统动作

- 创建测试任务对象
- 识别输入完整度
- 识别执行模式
- 识别是否需要额外补充信息

#### 输出

- 测试任务基础信息
- 测试模式判定
- 输入完整度评分

#### 完成标准

系统明确知道:

- 测什么
- 在哪测
- 以什么模式测
- 当前信息是否足以进入下一阶段

#### 失败与降级策略

- 缺 URL: 阻塞，不进入后续阶段
- 缺平台信息: 自动推断默认平台
- 缺文档: 降级到快速模式，不阻塞

### 6.2 阶段 2: 环境预检

#### 目标

在执行前明确系统是否具备足够的自动化能力。

#### 输入

- 测试任务
- 当前执行环境
- 执行器能力
- AI 与 OCR 能力

#### 系统动作

- 检查执行器是否在线
- 检查浏览器或设备是否可用
- 检查网络与证书状态
- 检查登录依赖、权限依赖、验证码风险
- 检查截图、控制台、网络采集能力

#### 输出

- 预检报告
- 自动化能力分级
- 阻塞项列表
- 可自动修复项列表

#### 完成标准

系统对本轮执行能力有明确判断:

- 可完整执行
- 可降级执行
- 暂不可执行

#### 失败与降级策略

- 执行器未启动: 自动拉起
- 浏览器异常: 自动重试初始化
- AI 不可用: 降级为规则引擎
- 网络采集不可用: 仅标记 API 维缺失，不阻塞其他维度
- 验证码不可自动通过: 进入 `blocked`，提示用户补充

### 6.3 阶段 3: 测试对象理解

#### 目标

让系统理解“当前被测对象在做什么，以及应该如何测试它”。

#### 输入

- URL
- 页面首屏内容
- 文档内容
- 历史知识库
- 平台信息

#### 系统动作

- 提取页面类型
- 提取页面主任务
- 提取主要操作入口
- 提取文档业务角色与流程
- 提取关键业务断言
- 提取风险点与约束条件

#### 输出

- 页面理解结果
- 文档理解结果
- 统一测试意图模型

#### 完成标准

系统能够回答以下问题:

- 当前页面或系统主要功能是什么
- 当前最值得测试的链路有哪些
- 哪些断言必须被验证

#### 失败与降级策略

- 文档解析失败: 保留页面理解路径继续
- 页面发现不完整: 以文档为主继续
- 页面与文档冲突: 记录冲突并生成高优先级验证项

### 6.4 阶段 4: 测试蓝图生成

#### 目标

把“理解结果”转换成可执行、可校验、可归因的测试蓝图。

#### 输入

- 测试意图模型
- 页面理解结果
- 文档理解结果
- 风险点

#### 系统动作

- 生成测试目标
- 拆分流程链路
- 生成步骤序列
- 绑定步骤断言
- 绑定步骤校验模板
- 给出优先级与执行顺序

#### 输出

- 测试蓝图
- 蓝图置信度
- 最小可执行集合
- 风险覆盖说明

#### 完成标准

蓝图中每一步都应明确:

- 做什么动作
- 为什么做
- 成功看什么
- 失败后重点检查什么

#### 失败与降级策略

- 理解结果不足: 生成最小冒烟蓝图
- 某些流程置信度低: 降低优先级，不阻塞整体执行
- 无法确认页面目标: 仅执行页面可用性与基础交互测试

### 6.5 阶段 5: 执行编排

#### 目标

把测试蓝图稳定转化为真实执行过程，并记录全过程上下文。

#### 输入

- 测试蓝图
- 执行环境
- 页面当前状态

#### 系统动作

- 导航
- 元素定位
- 动作执行
- 稳定性等待
- 异常恢复
- 快照采集
- 步骤结果记录

#### 输出

- 步骤执行记录
- 截图
- 控制台日志
- 网络快照
- 页面状态

#### 完成标准

每一步完成后，系统必须知道:

- 实际执行了什么
- 成功还是失败
- 当前页面处于什么状态
- 有哪些证据已被保存

#### 失败与降级策略

- 定位失败: 多级定位回退
- 页面抖动: 延迟等待并重试
- 弹窗阻塞: 自动识别与处理
- 新窗口跳转: 自动切换上下文
- 单步失败: 重试或局部跳过，保留失败证据
- 连续关键步骤失败: 中止当前链路并进入归因

### 6.6 阶段 6: 多维校验

#### 目标

对执行结果做统一质量判断，而不是单纯记录是否操作成功。

#### 输入

- 步骤执行记录
- 截图
- OCR 文本
- 控制台日志
- 网络快照
- 页面状态
- 业务断言

#### 系统动作

- UI 校验
- 控制台校验
- API 校验
- 业务校验
- 维度结果合并
- 证据对齐

#### 输出

- 步骤级多维校验结果
- 维度置信度
- 证据来源
- 可疑问题点列表

#### 完成标准

系统能够明确指出:

- 是动作失败，还是业务失败
- 是前端异常，还是接口异常
- 当前问题来自哪个维度，或是否是跨维度问题

#### 失败与降级策略

- OCR 不可用: 保留 DOM/UI 基础判断
- 控制台缺失: 仅降低 console 维可靠度
- 网络采集失败: 仅标记 API 维缺口
- 业务断言不足: 使用通用校验模板兜底

### 6.7 阶段 7: 缺陷归因

#### 目标

把多维异常信号聚合为可理解、可修复的缺陷对象。

#### 输入

- 多维校验结果
- 步骤记录
- 证据链
- 历史上下文

#### 系统动作

- 合并同源异常
- 判断严重等级
- 生成缺陷标题
- 构建现象描述
- 推测根因
- 生成修复建议
- 生成回归建议

#### 输出

- 缺陷对象
- 根因候选
- 因果链说明
- 修复上下文

#### 完成标准

每个缺陷至少能够回答:

- 发生了什么
- 在哪一步发生
- 为什么怀疑它发生
- 影响了什么
- 应优先检查哪里

#### 失败与降级策略

- 证据不足: 输出“候选根因”，不输出确定性结论
- 多个异常冲突: 拆成多个候选缺陷，并标记关联关系
- 无法合并因果链: 保留单点缺陷，避免错误归因

### 6.8 阶段 8: 结果交付

#### 目标

把整轮测试结果交付给不同消费方，并支持下一轮回归。

#### 输入

- 测试任务结果
- 执行记录
- 缺陷对象
- 修复上下文

#### 系统动作

- 生成任务摘要
- 生成执行报告
- 生成缺陷视图
- 生成结构化 JSON
- 生成 MCP 结果对象
- 生成回归入口信息

#### 输出

- 面向测试人员的结果视图
- 面向开发者的修复视图
- 面向 AI 助手的结构化输出
- 面向下一轮回归的最小验证集

#### 完成标准

结果交付后，消费方无需回到原始日志，就能完成下一步动作:

- 测试人员可确认问题范围
- 开发者可定位问题
- AI 助手可读取上下文
- 系统可基于缺陷生成回归任务

#### 失败与降级策略

- 报告渲染失败: 结构化输出仍应可用
- MCP 输出失败: 保留 JSON 报告
- 某些视图生成失败: 不影响核心缺陷数据落盘

## 7. 闭环中的关键对象模型

本文不定义代码级字段，只定义业务层对象责任。

### 7.1 TestTask

表示一次全自动测试任务。

建议包含:

- 目标对象
- 执行模式
- 阶段状态
- 自动化能力级别
- 当前结论

### 7.2 EnvironmentCheck

表示预检结果。

建议包含:

- 可用能力
- 阻塞项
- 可自动修复项
- 降级说明

### 7.3 UnderstandingResult

表示测试对象理解结果。

建议包含:

- 页面意图
- 文档意图
- 关键流程
- 风险点
- 必测断言

### 7.4 TestBlueprint

表示测试蓝图。

建议包含:

- 测试目标
- 链路集合
- 步骤集合
- 断言集合
- 优先级

### 7.5 StepRecord

表示一次步骤执行与校验结果。

建议包含:

- 动作
- 上下文
- 证据
- 多维判断
- 错误信息

### 7.6 Defect

表示一个可修复的缺陷对象。

建议包含:

- 标题
- 严重度
- 复现路径
- 证据链
- 根因候选
- 修复建议

### 7.7 RepairContext

表示交付给开发者与 AI 助手的修复上下文。

建议包含:

- 缺陷摘要
- 关键步骤
- 页面状态
- 控制台与网络证据
- 业务预期与实际偏差

## 8. 闭环状态机建议

### 8.1 任务状态

建议统一为以下状态:

- `draft`
- `prechecking`
- `understanding`
- `planning`
- `running`
- `analyzing`
- `completed`
- `completed_with_defects`
- `blocked`
- `cancelled`
- `error`

### 8.2 状态流转

```text
draft
  -> prechecking
  -> understanding
  -> planning
  -> running
  -> analyzing
  -> completed / completed_with_defects

任意阶段:
  -> blocked
  -> cancelled
  -> error
```

### 8.3 blocked 与 error 的区别

- `blocked`: 外部条件不足，系统知道为什么没法继续
- `error`: 系统内部异常，未能正常完成预期动作

这两者必须在产品层明确区分，否则用户很难理解问题归属。

## 9. 闭环中的自动化程度分级

为了评估 AutoTest 是否真的“全自动”，建议引入自动化程度分级。

### 9.1 分级建议

- `A0`: 仅完成入口收集，未进入执行
- `A1`: 完成预检，但需大量人工补充
- `A2`: 完成自动理解，生成最小蓝图
- `A3`: 自动执行完成，但归因较弱
- `A4`: 自动执行 + 自动校验 + 自动缺陷归因
- `A5`: 自动输出可直接被 AI 助手消费的修复上下文

### 9.2 用途

该分级可用于:

- 产品能力评估
- 执行结果解释
- 不同项目成熟度比较
- 后续路线图规划

## 10. 阶段级指标建议

闭环设计要落地，必须有阶段级指标，而不是只看最终通过率。

### 10.1 入口阶段指标

- URL 输入后任务创建成功率
- 输入信息完整度

### 10.2 预检阶段指标

- 自动修复成功率
- 阻塞识别准确率
- 降级执行比例

### 10.3 理解阶段指标

- 页面理解成功率
- 文档提取有效率
- 高风险点识别命中率

### 10.4 蓝图阶段指标

- 蓝图可执行率
- 高价值链路覆盖率
- 无需人工编辑蓝图比例

### 10.5 执行阶段指标

- 步骤成功率
- 自恢复成功率
- 平均单步耗时

### 10.6 校验与归因阶段指标

- 多维校验完整率
- 缺陷命中率
- 根因推断可采纳率

### 10.7 交付阶段指标

- 开发者消费率
- AI 助手消费成功率
- 修复后回归通过率

## 11. 闭环中的主要风险点

### 11.1 入口风险

- 用户输入过少，系统误判测试目标

### 11.2 预检风险

- 只提示问题，不做自动修复，导致“自动化”停留在表面

### 11.3 理解风险

- 文档与真实页面长期漂移，蓝图建立在过时规则上

### 11.4 蓝图风险

- 只生成“步骤”，不生成“断言”，最终退化成脚本回放

### 11.5 执行风险

- 过度依赖单一定位方式，导致大面积脆弱失败

### 11.6 校验风险

- 维度采集存在但没有真正融合，最终仍是多个孤立错误

### 11.7 归因风险

- 在证据不足时输出过强结论，误导开发修复方向

### 11.8 交付风险

- 报告对人可读，但对 AI 助手不可消费

## 14. 最终结论

AutoTest 后续所有设计，都应围绕一个简单而明确的问题:

**用户输入一个目标后，系统是否能够尽可能少依赖人工地完成测试，并交付可直接修复的问题结果。**

如果答案是“能”，说明闭环成立。
如果答案是“不能”，就说明还有阶段没有真正自动化，或者交付结果还不足以支撑修复。

因此，本文定义的八阶段闭环，应作为 AutoTest 后续产品文档、架构文档、接口文档和页面设计的统一基线。
