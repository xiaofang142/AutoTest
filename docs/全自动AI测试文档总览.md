## 3. 各文档职责

### 3.1 产品层

#### [全自动AI测试产品强化建议.md](./全自动AI测试产品强化建议.md)

回答:

- 为什么产品必须收束到“全自动 AI 测试”
- 哪些能力是主线，哪些能力是配套
- 产品后续该如何强化功能细节

#### [全自动AI测试闭环设计.md](./全自动AI测试闭环设计.md)

回答:

- 什么叫一次完整的全自动 AI 测试闭环
- 八个阶段分别做什么
- 每个阶段的输入、输出、完成标准和降级策略是什么

#### [自动测试任务模型设计.md](./自动测试任务模型设计.md)

回答:

- 产品主对象为什么应统一为 `TestTask`
- `Project`、`Run`、`Defect` 在新模型中的职责如何调整
- 前端、接口、状态机应如何围绕任务组织

#### [自动缺陷归因与AI交付设计.md](./自动缺陷归因与AI交付设计.md)

回答:

- 多维异常如何聚合成缺陷对象
- 缺陷证据链如何构建
- 如何生成开发者和 AI 助手都能直接消费的交付结果

### 3.2 工程层

#### [REQUIREMENTS.md](./REQUIREMENTS.md)

回答:

- 当前产品功能需求、非功能需求、技术约束是什么
- 哪些需求已经对齐到了任务驱动和闭环主线

#### [ARCHITECTURE.md](./ARCHITECTURE.md)

回答:

- 系统容器、模块与核心数据流如何支撑任务驱动闭环
- `TestTask`、`Run`、`Defect` 等对象在架构中的落位是什么

#### [API接口规范.md](./API接口规范.md)

回答:

- 当前接口规范如何从项目驱动逐步演进为任务驱动
- 推荐主线接口、状态模型和交付接口有哪些

## 4. 文档之间的关系

建议按以下依赖理解:

```text
产品强化建议
  -> 闭环设计
      -> 任务模型设计
      -> 缺陷归因与AI交付设计
          -> 需求规格说明书
          -> 架构设计
          -> API 接口规范
```

说明:

- 产品强化建议决定方向
- 闭环设计决定主线
- 任务模型和缺陷交付设计决定核心对象
- 需求、架构、接口负责工程化落地
