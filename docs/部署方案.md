## 2. 环境规划

### 2.1 环境分级

| 环境 | 用途 | PostgreSQL | AI 模型 | 执行器 |
|------|------|------------|---------|--------|
| dev | 本地开发 | Docker Local | gpt-4o-mini | 无/模拟 |
| test | 集成测试 | 独立实例 | gpt-4o-mini | 模拟执行器 |
| staging | 预发布 | 主从 | gpt-4o | 有限执行器(2) |
| production | 生产 | 主从+备份 | gpt-4o | 全量执行器集群 |

### 2.2 资源估算

```yaml
生产环境最小配置:
  API 节点: 2 台 4C/8G
  数据库: 1 台 8C/32G SSD 500G
  执行器（Web）: 2 台 8C/16G（每台可运行 4 个并发浏览器）
  Redis: 2C/4G
  对象存储: 500G

预估扩容:
  执行器数量 = (并发测试数 × 步数 × 步耗时) / (单机并发 × 目标完成时间)
  例: 100 个用例 × 10 步 × 5s / (4 并发 × 3600s) ≈ (5000) / 14400
  当并发测试 > 50 时，建议执行器节点 >= 3 台
```

## 3. 容器化部署 (Docker Compose)

### 3.1 docker-compose.yml

```yaml
version: "3.8"

services:
  api:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    env_file: .env
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
    restart: unless-stopped
    volumes:
      - ./app:/app/app  # dev only

  mcp:
    build:
      context: .
      dockerfile: Dockerfile
    command: uvicorn app.api.mcp.server:mcp_app --host 0.0.0.0 --port 8100
    ports:
      - "8100:8100"
    env_file: .env
    depends_on:
      - api
    restart: unless-stopped

  web-ui:
    build:
      context: ./web
      dockerfile: Dockerfile
    ports:
      - "3000:80"
    depends_on:
      - api
    restart: unless-stopped

  celery-worker:
    build:
      context: .
      dockerfile: Dockerfile
    command: celery -A app.workers.celery_app worker -l info -c 4
    env_file: .env
    depends_on:
      - db
      - redis
    restart: unless-stopped
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock  # 用于启动执行器容器

  celery-beat:
    build:
      context: .
      dockerfile: Dockerfile
    command: celery -A app.workers.celery_app beat -l info
    env_file: .env
    depends_on:
      - redis
    restart: unless-stopped

  db:
    image: postgres:15-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: autotest
      POSTGRES_USER: autotest
      POSTGRES_PASSWORD: autotest
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U autotest"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redisdata:/data
    restart: unless-stopped

  minio:
    image: minio/minio
    ports:
      - "9000:9000"
      - "9001:9001"
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    command: server /data --console-address ":9001"
    volumes:
      - miniodata:/data
    restart: unless-stopped

  executor-web:
    build:
      context: ./executor/web
      dockerfile: Dockerfile
    ports:
      - "3100:3100"
    environment:
      - MIDSCENE_MODEL_NAME=gpt-4o
      - PLAYWRIGHT_BROWSERS_PATH=/ms-playwright
    volumes:
      - /tmp/executor-screenshots:/screenshots
    shm_size: 2gb  # Chromium 需要共享内存
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: "4"
          memory: "8G"

volumes:
  pgdata:
  redisdata:
  miniodata:
```

### 3.2 Dockerfile (Python)

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# 系统依赖
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    && rm -rf /var/lib/apt/lists/*

# Python 依赖

RUN pip install --no-cache-dir -r requirements.txt

# 应用代码
COPY . .

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 5. 执行器集群部署

### 5.1 Web 执行器集群

```yaml
部署模式: 独立 Docker 容器 / K8s StatefulSet
每个节点:
  - 4C/8G 资源
  - 同时运行最多 4 个 Chromium 实例
  - 需要 2GB /dev/shm (共享内存)
  - 截图本地保存，异步上传到 S3

扩展策略:
  - 基于 Celery 队列长度自动扩缩
  - 每 100 个待执行用例增加 1 个执行器节点
  - 最大不超过 10 个节点

网络要求:
  - 能够访问被测应用
  - 能够访问 S3/MinIO
  - 能够与 API 服务通信
  - ~~不需要~~外部暴露端口（内部服务）
```

### 5.2 Android 执行器集群

```yaml
硬件要求:
  - 每节点连接 1-4 台 Android 设备
  - 或使用 Android 模拟器 (需要 KVM)
  - USB 连接或 ADB over TCP/IP

软件依赖:
  - ADB (Android Debug Bridge)
  - Android SDK Platform Tools
  - PaddleOCR + LLM analysis (via API)

部署方式:
  - 物理设备: 专用服务器 + USB Hub
  - 模拟器: Docker + Android Emulator (需要 KVM)
  
注意事项:
  - 设备管理: 需要设备池管理（哪里空闲分配哪里）
  - 设备重启: 每 N 次测试后重启设备清理状态
  - 截图存储: 本地缓存 + S3 长期存储
```

### 5.3 iOS 执行器集群

```yaml
硬件要求:
  - Mac mini / Mac Stuido (Apple Silicon)
  - 每台连接 1-2 台 iOS 设备
  - 或使用 iOS Simulator (仅 macOS)

软件依赖:
  - WebDriverAgent (Facebook)
  - ios-deploy
  - libimobiledevice
  - PaddleOCR + LLM analysis (via API)

注意事项:
  - 只有 macOS 可以运行 iOS 执行器
  - 需要 Apple Developer 证书
  - 设备自动解锁配置
  - 截图存储: 本地缓存 + S3 长期存储
```

## 6. 配置管理

### 6.1 配置层级

```yaml
配置加载顺序（后覆盖前）:
  1. 代码内默认值 (config.py)
  2. .env 文件 (开发环境)
  3. 环境变量 (所有环境)
  4. ConfigMap / Secret (K8s)
  5. 运行时配置中心 (Nacos / etcd)

配置分类:
  ├── 基础设施 (数据库、Redis、S3 连接)
  │   ├── 敏感: DATABASE_URL, REDIS_URL, S3_SECRET_KEY
  │   ├── 非敏感: S3_ENDPOINT, S3_BUCKET
  │
  ├── AI 服务 (模型、Key)
  │   ├── 敏感: LITELLM_API_KEY
  │   ├── 非敏感: EXTRACTION_MODEL, MAX_TOKENS
  │
  ├── 执行器 (超时、并发)
  │   ├── 非敏感: EXECUTOR_TIMEOUT, MAX_CONCURRENT
  │
  └── 应用配置 (保留天数、重试策略)
      └── 非敏感: RETENTION_DAYS, MAX_RETRIES
```

### 6.2 敏感信息管理

```yaml
原则:
  - 环境变量中的敏感值使用 Secret
  - 代码仓库中不存储任何敏感信息
  - 本地开发使用 .env (已加入 .gitignore)
  - 生产环境使用 K8s Secret 或 Vault

K8s Secret 示例:

  type: Opaque

    DATABASE_URL: "postgresql+asyncpg://..."

    S3_SECRET_KEY: "..."
```

## 7. CI/CD 流水线

### 7.1 流水线定义

```yaml
触发条件:
  - push to develop → 构建 + 测试 + 部署 staging
  - push to main → 构建 + 测试 + 部署 production
  - PR to develop → 构建 + 测试 (但不部署)

流水线阶段:
  ├── 1. Lint & Type Check
  │   ├── ruff format --check
  │   ├── ruff check
  │   └── mypy app/
  │
  ├── 2. Test
  │   ├── pytest tests/unit/ --cov --cov-fail-under=80
  │   └── pytest tests/integration/
  │
  ├── 3. Build
  │   ├── docker build -t autotest/api:$TAG
  │   ├── docker build -t autotest/web:$TAG
  │   └── docker build -t autotest/executor-web:$TAG
  │
  ├── 4. Staging Deploy (develop only)
  │   ├── docker compose -f docker-compose.staging.yml up -d
  │   └── run health check
  │
  └── 5. Production Deploy (main only)
      ├── tag release (vX.Y.Z)
      ├── rolling update K8s deployment
      └── smoke test
```

### 7.2 版本策略

```yaml
版本格式: v{major}.{minor}.{patch}
  - major: 不兼容的 API 变更
  - minor: 向下兼容的功能增
  - patch: Bug 修复

Tag 示例: v1.0.0, v1.1.0, v1.1.1

发布流程:
  develop → main 的 PR 合入后自动打 tag
  手动创建 Release Note
```

## 8. 监控与告警

### 8.1 基础设施监控

```yaml
指标采集: Prometheus + Node Exporter

关键指标:
  API:
    - 请求量 (QPS)
    - 响应延迟 (P50/P95/P99)
    - 错误率 (5xx / 4xx)
  
  数据库:
    - 连接数
    - 慢查询 (> 500ms)
    - 表空间使用率
  
  执行器:
    - 并发浏览器数
    - 任务队列长度
    - 步骤执行成功率
    - 平均步骤耗时
  
  系统:
    - CPU / 内存 / 磁盘使用率
    - 网络 I/O

告警规则:
  - API 错误率 > 1% (5分钟) → P1 告警
  - API P95 延迟 > 3s → P2 告警
  - 数据库连接 > 80% → P2 告警
  - 执行器队列 > 100 → P2 告警 (扩容)
  - 磁盘使用率 > 85% → P2 告警
```

### 8.2 日志收集

```yaml
收集: Filebeat → Elasticsearch 或 Loki
可视化: Kibana 或 Grafana

日志保留:
  - 活跃日志: 7 天 (ES 热节点)
  - 归档日志: 30 天 (ES 冷节点)
  - 持久日志: 1 年 (S3 冷存储)

关键日志查询:
  - request_id 追踪全链路日志
  - AI 调用记录（耗时、Token、模型）
  - 执行器步骤记录
```

### 8.3 看板

```yaml
Grafana 看板:
  1. API 概览: 请求量、延迟、错误率、活跃项目数
  2. 执行器状态: 队列深度、成功率、并发数、平均耗时
  3. AI 服务: Token 消耗、调用次数、延迟分布、成本估算
  4. 缺陷统计: 新增/修复趋势、严重级别分布、平台分布
  5. 系统资源: CPU/内存/磁盘趋势、容器重启次数
```

## 9. 灾备方案

### 9.1 数据库备份

```yaml
备份策略:
  全量备份: 每日 02:00 (pg_dump)
  增量备份: 每 30 分钟 (WAL 归档)
  
  保留策略:
    - 每日备份: 保留 30 天
    - 每周备份: 保留 3 个月
    - 每月备份: 保留 1 年

  备份验证:
    - 每日自动恢复测试
    - 周报包含备份状态

  恢复流程:
    1. 停止所有写操作
    2. 恢复到最近的全量备份
    3. 应用 WAL 到故障点
    4. 验证数据完整性
    5. 切换读写
```

### 9.2 服务容灾

```yaml
故障场景与应对:
  API 服务宕机:
    - K8s 自动重启
    - 多副本 + HPA 保障
    - 预计恢复: < 30s

  数据库宕机:
    - 自动切换到从库（只读模式）
    - 主库恢复后切换回来
    - 预计恢复: < 5min

  执行器宕机:
    - Celery 任务自动重调度到其他 Worker
    - 当前执行中的用例标记为 failed + 可重试
    - 预计恢复: < 1min (任务自动重调度)

  AI 服务不可用:
    - 自动切换到备用模型
    - 降级分析（跳过 LLM 判断部分）
    - 预计恢复: < 1min (自动切换)

  全站灾难:
    - 异地备份恢复
    - RPO: 24 小时
    - RTO: 4 小时
```

## 附录 A: Blue-Green 部署流程

### A.1 部署架构

```yaml
Blue-Green 部署:
  Blue: 当前生产环境
  Green: 新版本环境（与 Blue 完全相同的配置）

部署流程:
  1. 准备阶段:
     - 部署 Green 环境（与 Blue 相同的资源配置）
     - 运行数据库迁移（alembic upgrade）
     - 健康检查 Green 环境
  
  2. 切换阶段:
     - 更新负载均衡器指向 Green
     - 等待现有请求完成（最长 30s）
     - Green 开始接收新流量
  
  3. 验证阶段:
     - 监控 Green 错误率（5 分钟内 < 0.1%）
     - 运行 smoke test（3 个核心 API）
  
  4. 清理阶段:
     - 保持 Blue 环境 24 小时
     - 24 小时后无问题 → 回收 Blue 环境
     - 如有问题 → 切回 Blue（回滚）

切换时间:
  - 预期: < 10 秒（DNS/负载均衡切换时间）
  - 最大: 30 秒（等待正在执行的 API 请求完成）

数据库兼容性:
  - 数据库变更必须向后兼容（旧代码可读新 schema）
  - 需要数据迁移的变更：分两步部署
    - 第一步: 添加新字段（Blue + Green 都兼容）
    - 第二步: 迁移数据 + 移除旧字段
```

### A.2 Blue-Green 切换脚本

```bash
#!/bin/bash
# blue-green-switch.sh

ENV=${1:-production}
BLUE_UPSTREAM="blue-autotest-api:8000"
GREEN_UPSTREAM="green-autotest-api:8000"

echo "=== Blue-Green Deploy: $ENV ==="

# 1. 健康检查 Green
echo "→ Health check Green..."
curl -f http://$GREEN_UPSTREAM/health || exit 1

# 2. 运行 smoke test
echo "→ Running smoke tests..."
curl -f http://$GREEN_UPSTREAM/api/v1/projects?page=1 || exit 1

# 3. 切换负载均衡
echo "→ Switching traffic to Green..."
# Nginx: 更新 upstream 配置
sed -i "s/$BLUE_UPSTREAM/$GREEN_UPSTREAM/g" /etc/nginx/upstreams.conf
nginx -s reload

# 4. 等待旧连接耗尽
echo "→ Waiting for old connections to drain..."
sleep 10

# 5. 验证新环境
echo "→ Verifying Green..."
for i in {1..5}; do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost/health)
  if [ "$STATUS" == "200" ]; then
    echo "✓ Green is healthy after switch"
    break
  fi
  sleep 2
done

echo "=== Blue-Green switch complete ==="
echo "Blue kept for 24h rollback: $BLUE_UPSTREAM"
```

## 附录 C: Rollback 流程

### C.1 数据库回滚

```yaml
场景: 数据库迁移导致问题

回滚步骤:
  1. 立即停止新迁移（正在执行的迁移）
  2. 执行 alembic downgrade 回到前一版本
  3. 验证降级后的 schema 与应用代码兼容
  4. 如果降级失败（数据丢失不可逆）:
     a. 从备份恢复数据库
     b. 接受最多 24 小时的数据丢失（RPO）
  
  回滚命令:
    alembic downgrade -1          # 回滚一步
    alembic downgrade <revision>  # 回滚到指定版本
  
  预防措施:
    - 所有迁移必须实现 downgrade()
    - 大表迁移使用先加字段后移除策略
    - 生产迁移前先在 staging 执行
```

### C.2 代码回滚

```yaml
Blue-Green 环境回滚:
  1. 将负载均衡指回 Blue 环境
  2. 验证 Blue 环境健康
  3. 记录回滚原因和影响范围
  4. 保留 Green 环境用于问题分析

Docker Compose 回滚:
  docker compose down
  docker compose -f docker-compose.rollback.yml up -d

K8s 回滚:
  kubectl rollout undo deployment/autotest-api
  kubectl rollout status deployment/autotest-api

回滚验证:
  - API 健康检查
  - 数据库连接正常
  - Celery Worker 正常连接
  - 执行器通信正常
```
