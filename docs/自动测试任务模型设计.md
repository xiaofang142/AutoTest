## 1. 设计目标

为了让 AutoTest 真正围绕“全自动 AI 测试”运转，需要一个统一的一等对象来承载:

- 用户输入
- 闭环阶段
- 自动化能力
- 测试蓝图
- 执行结果
- 缺陷交付

这个一等对象建议定义为:

**`TestTask`**

本文档的目标是:

1. 统一产品、架构、接口、前端页面所使用的任务对象
2. 让“创建项目、上传文档、生成场景、创建执行”等内部动作，被收敛为任务生命周期的一部分
3. 让用户看到的主线变成“自动测试任务”，而不是若干离散技术对象

---

## 2. 为什么需要任务模型

当前文档与系统设计中，主流程通常被拆为:

```text
项目 -> 文档 -> 知识库 -> 场景 -> 执行 -> 报告 -> 缺陷
```

这条链路对系统内部实现有价值，但对用户和消费方不够自然。

用户真实关注的是:

```text
我要测什么
-> 现在测到哪一步了
-> 测完了吗
-> 发现了什么问题
-> 我该怎么修
```

因此需要一个上层业务对象，把这些内部模块统一包裹起来。

## 3. 任务模型的定位

### 3.1 TestTask 是产品主对象

建议后续产品主视角统一为 `TestTask`。

它表示一次完整的自动测试尝试，从输入目标开始，到最终交付结构化结果为止。

### 3.2 Project 退居归档容器

`Project` 仍可以存在，但其职责建议调整为:

- 保存长期配置
- 管理文档资产
- 承载多次任务历史
- 作为统计和归档维度

用户在日常使用时，优先面向 `TestTask`，而不是优先面向 `Project`。

### 3.3 Run 退居执行子对象

`Run` 不再是顶层业务入口，而是 `TestTask` 在执行阶段产出的一个子对象。

关系建议如下:

```text
Project
  └── TestTask
        ├── TaskInput
        ├── EnvironmentCheck
        ├── UnderstandingResult
        ├── TestBlueprint
        ├── ExecutionRun
        ├── Defect[]
        └── DeliveryPackage
```

## 4. 顶层对象图

```text
TestTask
├── 基础信息
├── 输入信息
├── 阶段状态
├── 自动化能力
├── 预检结果
├── 理解结果
├── 测试蓝图
├── 执行结果
├── 缺陷汇总
└── 交付结果
```

建议把 `TestTask` 设计成“聚合根”。

所有与一次自动测试强相关的结构，都应可通过任务对象被追溯和访问。

## 5. TestTask 模型定义

以下定义是业务模型，不限定代码字段名，但建议后续接口与前端结构保持一致。

### 5.1 基础信息

#### 作用

标识任务身份、归属、创建背景。

#### 建议字段

| 字段 | 说明 |
|---|---|
| `task_id` | 任务唯一标识，建议前缀 `task_` |
| `project_id` | 可选，表示归属项目 |
| `name` | 任务名称 |
| `description` | 任务说明 |
| `source` | 任务来源，例如 Web UI / CLI / API / MCP |
| `created_by` | 发起方 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 5.2 目标信息

#### 作用

描述这次任务到底要测什么。

#### 建议字段

| 字段 | 说明 |
|---|---|
| `target_type` | Web / H5 / Android / iOS |
| `target_url` | Web/H5 入口地址 |
| `target_app` | App 包名、活动页或深链入口 |
| `environment` | dev / test / staging / prod-like |
| `scope` | 全站 / 指定模块 / 指定页面 / 指定链路 |

### 5.3 模式信息

#### 作用

描述任务以什么自动化深度运行。

#### 建议字段

| 字段 | 说明 |
|---|---|
| `mode` | `quick` / `document_driven` / `mixed` |
| `goal` | 冒烟 / 回归 / 指定业务 / 缺陷复测 |
| `depth` | `minimal` / `standard` / `deep` |
| `auto_level` | 自动化等级，见后文 |

### 5.4 状态信息

#### 作用

描述任务当前处于闭环的哪个阶段。

#### 建议字段

| 字段 | 说明 |
|---|---|
| `status` | 当前状态 |
| `current_stage` | 当前阶段名称 |
| `stage_started_at` | 当前阶段开始时间 |
| `progress_percent` | 进度百分比 |
| `blocked_reason` | 阻塞原因 |
| `error_summary` | 错误摘要 |

### 5.5 结果摘要

#### 作用

快速描述任务最终结论。

#### 建议字段

| 字段 | 说明 |
|---|---|
| `final_status` | `completed` / `completed_with_defects` / `blocked` / `error` |
| `summary` | 任务摘要 |
| `defect_count` | 缺陷数 |
| `high_risk_count` | 高风险问题数 |
| `delivery_ready` | 是否已形成可交付结果 |

## 6. 子对象定义

### 6.1 TaskInput

#### 作用

保存用户显式输入与系统自动补全信息。

#### 建议包含

- URL / App 入口
- 文档列表
- 账号信息
- 目标范围
- 优先测试目标
- 平台配置
- 用户显式备注

#### 设计原则

- 明确区分“用户输入”和“系统推断”
- 不要求用户理解内部对象
- 能默认的尽量默认，减少输入负担

### 6.2 EnvironmentCheck

#### 作用

记录任务在预检阶段对环境与能力的判断。

#### 建议包含

- 执行器状态
- 浏览器或设备状态
- 网络状态
- 登录前置依赖
- OCR 状态
- AI 状态
- 日志采集能力
- 网络采集能力
- 可自动修复项
- 阻塞项

#### 设计原则

- 不是简单做健康检查，而是对“本轮是否可以自动跑通”做结论

### 6.3 UnderstandingResult

#### 作用

记录系统对被测对象和测试意图的理解。

#### 建议包含

- 页面意图
- 文档意图
- 关键角色
- 关键流程
- 风险点
- 必测断言
- 页面对象清单
- 文档与页面冲突点

#### 设计原则

- 从“规则提取”升级为“测试意图理解”

### 6.4 TestBlueprint

#### 作用

记录系统自动生成的测试蓝图。

#### 建议包含

- 测试目标集合
- 流程链路集合
- 步骤序列
- 步骤断言
- 步骤优先级
- 风险覆盖说明
- 蓝图置信度

#### 设计原则

- 不只是步骤集合，还必须包含断言与诊断意图

### 6.5 ExecutionRun

#### 作用

记录蓝图被真实执行后的运行对象。

#### 建议包含

- 浏览器或设备上下文
- 执行起止时间
- 步骤执行记录
- 当前页面状态
- 执行时错误
- 重试记录
- 恢复动作记录

#### 设计原则

- `Run` 只负责执行过程，不承担完整任务上下文

### 6.6 DefectSummary

#### 作用

从任务角度汇总本轮缺陷情况。

#### 建议包含

- 缺陷数量
- 严重度分布
- 模块分布
- 影响链路
- 是否存在阻断性问题

### 6.7 DeliveryPackage

#### 作用

保存最终交付给不同消费方的结果包。

#### 建议包含

- 测试人员视图
- 开发者视图
- AI 助手结构化结果
- 回归验证入口

## 7. 任务生命周期定义

### 7.1 标准状态

建议统一任务状态如下:

- `draft`
- `prechecking`
- `understanding`
- `planning`
- `running`
- `analyzing`
- `completed`
- `completed_with_defects`
- `blocked`
- `cancelled`
- `error`

### 7.2 状态语义

| 状态 | 说明 |
|---|---|
| `draft` | 已创建但未开始正式处理 |
| `prechecking` | 正在做环境与能力预检 |
| `understanding` | 正在理解页面、文档与测试意图 |
| `planning` | 正在生成测试蓝图 |
| `running` | 正在执行测试步骤 |
| `analyzing` | 正在做多维校验与缺陷归因 |
| `completed` | 全流程完成且无缺陷 |
| `completed_with_defects` | 全流程完成但发现缺陷 |
| `blocked` | 因外部条件不足被阻塞 |
| `cancelled` | 被用户或系统取消 |
| `error` | 系统内部异常导致失败 |

### 7.3 状态流转

```text
draft
  -> prechecking
  -> understanding
  -> planning
  -> running
  -> analyzing
  -> completed / completed_with_defects

任意阶段:
  -> blocked
  -> cancelled
  -> error
```

### 7.4 blocked 与 error 的处理差异

- `blocked`: 需要向用户说明“缺什么、为什么缺、补什么”
- `error`: 需要向内部说明“系统哪里出错了、影响范围是什么”

## 8. 任务与内部模块的映射关系

为了兼容现有系统实现，建议统一映射如下:

| 内部模块 | 对应任务子对象 |
|---|---|
| 项目配置 | `TestTask.project_id` + 归档信息 |
| 文档管理 | `TaskInput.documents` |
| 文档解析 | `UnderstandingResult.document_understanding` |
| 页面发现 | `UnderstandingResult.page_understanding` |
| 场景生成 | `TestBlueprint` |
| 执行记录 | `ExecutionRun` |
| 缺陷列表 | `DefectSummary` + `Defect[]` |
| 报告输出 | `DeliveryPackage` |

这样可以做到:

- 外部以任务驱动
- 内部仍能复用既有模块

## 9. 快速模式与文档驱动模式在任务中的体现

### 9.1 quick 模式

特点:

- 只有目标地址
- 页面理解优先
- 蓝图较小
- 冒烟验证优先

在 `TestTask` 中表现为:

- `mode = quick`
- `documents = []`
- `understanding_result` 以页面理解为主
- `blueprint` 以通用校验模板为主

### 9.2 document_driven 模式

特点:

- 有文档
- 文档与页面联合理解
- 蓝图更偏业务链路
- 归因深度更强

在 `TestTask` 中表现为:

- `mode = document_driven`
- `documents` 非空
- `understanding_result` 包含业务规则
- `blueprint` 包含业务断言与高风险路径

### 9.3 mixed 模式

特点:

- 有部分文档，但不完整
- 同时依赖页面发现与文档理解
- 是实际最常见的企业场景

建议在产品层明确支持 `mixed` 模式，而不是强行二分。

## 10. 任务自动化等级定义

建议把自动化程度也挂在任务对象上。

### 10.1 等级定义

- `A0`: 仅收集输入
- `A1`: 完成预检，但需大量人工补充
- `A2`: 已生成最小可执行蓝图
- `A3`: 已自动执行完成
- `A4`: 已自动完成执行、校验、缺陷归因
- `A5`: 已形成可直接供 AI 助手消费的修复上下文

### 10.2 用途

- 解释本轮任务为何成功或失败
- 体现产品成熟度
- 支持多轮结果对比

## 11. 前端视图建议

任务模型落地后，前端应围绕任务组织页面。

### 11.1 首页

建议首页主入口为:

- 输入目标地址
- 选择平台
- 上传文档
- 选择测试深度
- 一键创建任务

### 11.2 任务列表

建议展示:

- 任务名称
- 目标对象
- 当前阶段
- 自动化等级
- 最终结果
- 缺陷数
- 创建时间

### 11.3 任务详情

建议拆为标签页:

- 概览
- 输入信息
- 预检结果
- 理解结果
- 测试蓝图
- 执行过程
- 缺陷与交付

## 12. API 设计建议

后续接口应逐步以任务对象为中心。

### 12.1 推荐接口

- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `GET /api/v1/tasks/{task_id}`
- `POST /api/v1/tasks/{task_id}/start`
- `POST /api/v1/tasks/{task_id}/cancel`
- `GET /api/v1/tasks/{task_id}/timeline`
- `GET /api/v1/tasks/{task_id}/delivery`

### 12.2 子资源接口

- `GET /api/v1/tasks/{task_id}/environment-check`
- `GET /api/v1/tasks/{task_id}/understanding`
- `GET /api/v1/tasks/{task_id}/blueprint`
- `GET /api/v1/tasks/{task_id}/run`
- `GET /api/v1/tasks/{task_id}/defects`

### 12.3 兼容策略

旧接口可以暂时保留:

- `/projects/*`
- `/runs/*`
- `/defects/*`

但推荐在文档中明确它们属于底层兼容接口，而不是推荐主线接口。

## 13. WebSocket / 事件模型建议

任务模型下，实时事件建议按任务阶段组织。

### 13.1 阶段事件

- `task_created`
- `task_precheck_started`
- `task_precheck_completed`
- `task_understanding_started`
- `task_understanding_completed`
- `task_planning_started`
- `task_planning_completed`
- `task_execution_started`
- `task_step_completed`
- `task_analysis_started`
- `task_defect_found`
- `task_completed`
- `task_blocked`
- `task_cancelled`
- `task_error`

### 13.2 事件载荷建议

每个事件建议至少带:

- `task_id`
- `stage`
- `event_type`
- `timestamp`
- `summary`
- `progress`
- `data`

## 14. 与缺陷模型的关系

`TestTask` 不负责完整定义缺陷对象，但必须能承载缺陷交付入口。

任务层至少应知道:

- 本轮缺陷总数
- 高风险缺陷数
- 关键阻断缺陷
- 缺陷详情入口
- 交付包入口

详细缺陷模型建议在独立文档中定义。

---

## 15. 结论

`TestTask` 是当前文档体系中的任务主对象。

本文只负责定义:

- 为什么任务应成为产品主视角
- 任务聚合哪些核心子对象
- 任务如何承载状态、执行与交付
