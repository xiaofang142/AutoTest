## 1. 本轮讨论结论

本轮建议统一收束到一个核心方向:

**AutoTest 不是一个通用测试管理平台，而是一个围绕"全自动 AI 测试"闭环构建的产品。**

因此，后续所有需求、架构、接口、前端页面、执行器能力、报告设计，都建议围绕以下唯一主线展开:

```text
输入目标 -> 自动理解 -> 自动生成测试 -> 自动执行 -> 自动校验 -> 自动归因 -> 自动交付可修复结果
```

如果某个模块不能直接强化这条链路，就不应在当前阶段占据产品中心位置。

## 2. 产品主轴重定义

### 2.1 一句话定义

用户只需要提供被测对象地址，以及可选的 PRD / UI 规范 / API 文档，系统即可自动完成测试准备、场景生成、执行、诊断与报告输出。

### 2.2 产品北极星

北极星不应是"项目数"、"场景数"、"缺陷数"，而应是以下三项:

1. **从输入地址到产出首份报告的总耗时**
2. **无需人工编辑即可完成的测试比例**
3. **缺陷报告被开发者或 AI 助手直接消费并修复的成功率**

### 2.3 产品边界建议

当前阶段建议把 AutoTest 明确聚焦为:

- 全自动 UI 测试
- 自动场景生成
- 自动多维校验
- 自动缺陷归因
- 面向开发者与 AI 助手的结构化输出

当前阶段不建议把产品重心放在:

- 复杂项目管理
- 多租户权限系统
- 重型测试资产运营
- 非核心的报表 BI
- 通用平台治理能力

这些能力可以存在，但都应退居配套地位。

## 3. 全流程重构原则

全流程的详细阶段定义已经在 [全自动AI测试闭环设计.md](./全自动AI测试闭环设计.md) 中完整收敛，这里只保留产品层必须坚持的重构原则。

### 3.1 唯一主线

产品主线固定为:

```text
输入目标 -> 自动理解 -> 自动生成测试 -> 自动执行 -> 自动校验 -> 自动归因 -> 自动交付可修复结果
```

如果某个功能不能直接强化这条链路，就不应成为当前版本的产品中心。

### 3.2 产品层只保留四个关键决策

1. 首页主入口改为“新建自动测试任务”，而不是“新建项目”
2. 项目退为归档容器，任务成为主视角
3. 场景生成升级为测试蓝图生成
4. 报告升级为面向开发者与 AI 助手的结构化交付

- 八阶段闭环与降级策略: [全自动AI测试闭环设计.md](./全自动AI测试闭环设计.md)

## 4. 功能细节强化建议

下面按产品功能域给出建议，所有功能都应直接服务"全自动 AI 测试闭环"。

### 4.1 测试任务中心

建议新增并强化"测试任务"作为一等对象。

任务应包含:

- 目标系统
- 测试模式
- 输入资料
- 执行策略
- 当前阶段
- 自动化程度
- 最终报告

建议任务状态统一为:

- `draft`
- `prechecking`
- `understanding`
- `planning`
- `running`
- `analyzing`
- `completed`
- `completed_with_defects`
- `blocked`
- `cancelled`

### 4.2 自动化程度可视化

建议引入"自动化完成度"概念，而不是简单显示成功/失败。

可拆为:

- 自动准备完成度
- 自动理解完成度
- 自动执行完成度
- 自动诊断完成度
- 需人工介入点数量

这样可以直接体现产品是否真正做到"全自动"。

### 4.3 文档理解中心

文档模块不应只是上传与解析记录，应升级为:

- 文档来源管理
- 文档版本管理
- 文档可信度管理
- 规则提取结果确认
- 测试蓝图引用关系

建议新增的核心视图:

- 文档摘要
- 提取规则
- 风险点清单
- 与场景/蓝图的映射

### 4.4 自动生成策略中心

建议把生成能力从"生成场景"扩展为:

- 生成基础冒烟集
- 生成核心业务链路集
- 生成高风险异常集
- 生成权限与角色差异集
- 生成回归最小集

用户应该只需要选择:

- 测什么
- 测到多深
- 接受多长耗时

剩余由系统自动决定。

### 4.5 执行过程透明化

全自动不代表黑盒。

建议执行页明确展示:

- 当前阶段
- 当前页面
- 当前步骤
- 系统为何执行这一步
- 该步骤正在验证什么
- 若失败，系统准备怎么诊断

这能显著增强用户对 AI 测试过程的信任感。

### 4.6 缺陷对象升级

缺陷对象建议从"报告条目"升级为"可回归验证的修复任务单元"。

每个缺陷建议具备:

- 严重级别
- 归属模块
- 复现链路
- 证据链
- 根因推断
- 修复建议
- 回归入口
- 对 AI 助手的结构化上下文

### 4.7 面向 AI 助手的交付模型

建议把"结构化参考数据"能力前置到产品设计中心。

输出不应只是:

- `test failed`

而应至少包括:

- 失败步骤
- 页面截图
- OCR 文本
- 控制台错误
- 网络请求摘要
- 关键 DOM / 页面状态
- 业务断言结果
- 根因候选
- 修复建议

## 5. 前端信息架构建议

为了让产品真正围绕全自动 AI 测试运转，建议前端导航重构为以下顺序:

1. **新建自动测试**
2. **测试任务**
3. **执行详情**
4. **缺陷中心**
5. **知识/文档**
6. **系统设置**

不建议继续以"项目管理"作为最强入口。

### 5.1 首页建议

首页第一屏建议直接提供:

- 输入测试地址
- 上传或粘贴文档
- 选择测试模式
- 一键开始

首页不应先展示大量运营式统计卡片，因为这会削弱产品的即时价值感。

### 5.2 执行详情页建议

执行详情页建议按以下结构展示:

- 本轮结论
- 任务进度
- 步骤流水
- 失败节点
- 缺陷因果链
- 证据链
- AI 修复建议

### 5.3 缺陷详情页建议

缺陷详情页建议使用固定模板:

- 缺陷标题
- 严重程度
- 影响功能
- 复现步骤
- 页面截图
- 控制台证据
- 网络证据
- 根因分析
- 修复建议
- 回归建议

## 8. 本轮建议的优先级

本轮只讨论文档时，建议优先明确以下内容:

### P0

- 产品是否彻底以"全自动 AI 测试任务"为主视角
- 全流程阶段定义是否统一
- 自动化边界与降级策略是否明确
- 缺陷交付模型是否面向开发者与 AI 助手

### P1

- 前端导航与页面信息架构是否调整
- 场景生成是否升级为测试蓝图生成
- 实时事件模型是否统一

### P2

- 指标体系与运营看板如何围绕自动化效果重构
- 多平台统一体验如何进一步抽象

## 10. 最终判断

AutoTest 当前最值得强化的不是"再加一些测试功能"，而是把所有能力重新压到一条真正的产品主线上:

**让用户输入目标后，系统能够自动完成测试并交付可直接修复的问题结果。**

只要文档、产品结构、接口模型、前端视图都持续围绕这条主线演进，项目才会从"功能集合"变成"真正的全自动 AI 测试产品"。
