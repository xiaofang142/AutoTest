## 1. 开发环境搭建

### 1.1 系统要求

```yaml
操作系统:
  - macOS 14+ (开发)
  - Ubuntu 22.04+ (服务器)
  - Windows 11 + WSL2 (可选)

Python: 3.11+
Node.js: 20+ (仅执行器)
Docker: 24+ (含 Compose V2)
PostgreSQL: 15+
Redis: 7+
```

### 1.2 一键启动

```bash
# 1. 克隆仓库
git clone https://github.com/org/autotest.git
cd autotest

# 2. 创建虚拟环境
python -m venv .venv
source .venv/bin/activate

# 3. 安装依赖
pip install -r requirements-dev.txt

# 4. 配置环境变量
cp .env.example .env
# 编辑 .env，填入必要的配置（数据库 URL、AI API Key 等）

# 5. 启动基础设施
docker compose up -d db redis

# 6. 初始化数据库
alembic upgrade head

# 7. 启动服务
uvicorn app.main:app --reload --port 8000

# 8. 可选：启动执行器
cd executor/web && npm install && npm run dev
```

### 1.3 环境变量


## 3. 编码规范

### 3.1 Python 编码规范

```yaml
语言版本: Python 3.11+
格式化: Ruff (取代 Black + isort)
  命令: ruff format .
  line-length: 100

类型检查: mypy (strict mode)
  配置文件: pyproject.toml 中的 [tool.mypy]

Linting:
  - Ruff: 所有规则启用
  - 命令: ruff check .
  - 自动修复: ruff check --fix .

命名规范:
  模块: snake_case (project_service.py)
  类: PascalCase (ProjectService)
  函数/方法: snake_case (create_project)
  变量: snake_case (project_id)
  常量: UPPER_SNAKE_CASE (MAX_RETRY_COUNT)
  私有成员: _前缀 (_internal_method)
  # 类型别名: PascalCase + Type 后缀 (ProjectIDType)

导入顺序:
  1. 标准库 (os, sys, datetime)
  2. 第三方库 (fastapi, sqlalchemy)
  3. 项目内部 (app.domain, app.services)
  每组内按字母序排列

异步:
  - 所有 I/O 操作使用 async/await
  - 耗时 CPU 操作使用 run_in_executor
  - 避免在异步代码中使用 time.sleep()
  - 使用 asyncio.gather() 并行执行独立任务
```

### 3.2 TypeScript 编码规范

```yaml
语言版本: TypeScript 5.0+
格式化: Prettier
  tab-width: 2
  semi: true
  single-quote: true

Linting: ESLint + @typescript-eslint

命名规范:
  文件: camelCase (executor.ts)
  类: PascalCase (WebExecutor)
  接口: PascalCase + I 前缀 (IExecutorClient)
  函数: camelCase (executeStep)
  变量: camelCase (stepResult)
  常量: UPPER_SNAKE_CASE (MAX_RETRY_COUNT)

异步:
  - 所有 I/O 使用 async/await
  - 避免 callback hell
  - 使用 Promise.all() 并行执行
```

### 3.3 SQL 规范

```yaml
命名:
  表名: snake_case 复数 (projects, step_records)
  列名: snake_case (created_at, project_id)
  主键: id (统一为 VARCHAR(32))
  外键: 单数表名_id (project_id, run_id)
  索引: idx_{表}_{列}
  唯一约束: uq_{表}_{列}

最佳实践:
  - 使用 EXPLAIN ANALYZE 验证查询性能
  - CTE 优于子查询
  - 避免 SELECT *，明确列出需要的列
  - 所有表必须有 created_at 和 updated_at
  - 大表必须有合适的索引
  - 避免在 WHERE 子句中对列使用函数
```

## 4. Git 工作流

### 4.1 分支策略

```yaml
分支模型: GitHub Flow (简化)

分支类型:
  main:      生产分支，保持可部署状态
  develop:   开发分支，功能集成
  feat/*:    功能开发 (feat/add-executor-client)
  fix/*:     缺陷修复 (fix/step-record-timeout)
  docs/*:    文档变更 (docs/api-spec-v2)
  refactor/*:重构 (refactor/analysis-engine)

规则:
  - 永远不要直接提交到 main
  - feat/* 从 develop 创建，合回 develop
  - develop 合并到 main 时发布新版本
  - 每个 PR 必须通过 CI 和代码审查
```

### 4.2 提交规范

```yaml
格式: Conventional Commits

类型:
  feat:     新功能 (feat: add project creation API)
  fix:      Bug 修复 (fix: fix null pointer in step analysis)
  docs:     文档变更 (docs: update API spec)
  refactor: 重构 (refactor: extract cross-dimension analyzer)
  test:     测试 (test: add unit tests for document parser)
  chore:    维护 (chore: update dependencies)

示例:
  feat(project): add document reference management
  fix(analysis): handle null value in evidence chain
  docs(api): add MCP interface documentation
  refactor(executor): extract base executor class
```

### 4.3 PR 规范

```yaml
PR 标题: [类型] 简短描述 (#issue_number)
PR 描述模板:
  ## 变更内容
  - 做了什么
  - 为什么这样做
  
  ## 相关 Issue
  Closes #xxx
  
  ## 测试验证
  - [ ] 单元测试通过
  - [ ] 集成测试通过
  - [ ] 手动测试通过
  
  ## Review 重点
  - 需要特别关注的代码

PR 大小: 建议 < 400 行变更
Reviewer: 至少 1 人
```

## 5. 代码审查规范

### 5.1 Review 要点

```yaml
必须检查:
  □ 架构层是否正确（接口层 vs 业务编排层 vs 领域层）
  □ 模块间依赖方向是否正确
  □ 领域层是否有外部框架侵入
  □ 错误处理是否覆盖了所有外部调用
  □ 异步边界是否合理（哪些用 Celery，哪些同步）
  □ 敏感数据是否已脱敏
  □ 日志是否在关键节点记录
  □ 是否需要数据库迁移
  □ API 变更是否已更新文档
  □ 测试覆盖是否 >= 80%
```

### 5.2 合入门禁

```yaml
CI 检查:
  □ ruff format --check
  □ ruff check
  □ mypy --strict
  □ pytest (coverage >= 80%)
  □ 构建镜像验证

必须通过:
  - 所有 CI 检查
  - 至少 1 人 Review 通过
  - 无未解决 conversation

例外: 紧急修复可跳过 Review，但合入后 24 小时内必须补 Review
```

## 6. 文档规范

### 6.1 文档责任

```yaml
每个模块的文档责任:
  - 接口层: 路由定义、请求/响应格式、错误码
  - 业务编排层: 核心流程、事务边界
  - 领域层: 实体定义、值对象、业务规则
  - 基础设施层: 配置说明、第三方集成

文档更新时机:
  - API 变更时同步更新 API 文档
  - 架构变更时更新架构文档
  - 数据库变更时同步更新 ER 图
  - 每次 PR 合入后检查关联文档是否需要更新
```

### 6.2 代码注释规范

```yaml
公共 API 必须有 docstring:
  def create_project(name: str, platforms: list[str]) -> Project:
      """创建测试项目。
      
      Args:
          name: 项目名称，不能为空
          platforms: 目标平台列表，至少一个
      
      Returns:
          创建成功的项目实体
      
      Raises:
          InvalidParameterError: 参数校验失败
      """
      ...

复杂逻辑必须有行内注释:
  # 多策略并行提取：同一内容用 4 种不同 Prompt 提取
  # 合并时取交集为高置信度，并集减交集为候选
  ...

避免的注释:
  - 显而易见的注释（# 创建项目）
  - 过时的注释（修改代码后必须同步更新注释）
  - 注释掉的代码（直接删除，git 历史中可追溯）
```

## 7. 质量门禁

### 7.1 本地开发检查

```bash
# 提交前运行
ruff format .           # 格式化
ruff check --fix .      # Lint + 自动修复
mypy app/               # 类型检查
pytest tests/           # 测试
```

### 7.2 CI 流水线

```yaml
阶段:
  lint:
    - ruff format --check
    - ruff check
    - mypy app/
  
  test:
    - pytest tests/unit/ --cov=app --cov-fail-under=80
    - pytest tests/integration/
  
  build:
    - docker build -t autotest-api .
    - docker build -t autotest-executor ./executor/web
  
  deploy:
    - docker compose -f docker-compose.yml up -d
```

### 7.3 覆盖率要求

```yaml
总体目标: >= 80%
单元测试:
  领域层: >= 95%
  业务编排层: >= 85%
  接口层: >= 80%
  基础设施层: >= 70%

免测范围:
  - 配置加载类代码（config.py）
  - 纯数据类（dataclass 实体）
  - 外部 SDK 包装层（简单代理）
```

### C.3 Flag 生命周期管理

```python
class FeatureFlag:
    """Feature Flag 管理器"""
    
    @classmethod
    async def is_enabled(cls, flag_name: str, default: bool = False) -> bool:
        """检查 Flag 是否启用"""
        config = await DynamicConfig.get(f"feature.{flag_name}")
        if config is None:
            return default
        return config.get("enabled", default)
    
    @classmethod
    async def get_variant(cls, flag_name: str, default: str = None) -> str:
        """获取 Flag 的变体值"""
        config = await DynamicConfig.get(f"feature.{flag_name}")
        if config is None:
            return default
        return config.get("variant", default)
    
    @classmethod
    async def set_flag(cls, flag_name: str, enabled: bool, 
                       reason: str = "", owner: str = "") -> None:
        """设置 Flag 状态"""
        await DynamicConfig.set(
            f"feature.{flag_name}",
            {"enabled": enabled, "reason": reason, 
             "owner": owner, "updated_at": datetime.now().isoformat()}
        )

# 注册所有 Flag（用于清理检查）
ALL_FEATURE_FLAGS = {

        "owner": "platform-team",
        "created_at": "2026-05-14",

    },

}
```
