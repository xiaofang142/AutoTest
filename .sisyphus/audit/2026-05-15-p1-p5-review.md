## Oracle 架构审查 — 深度发现

### Bug A: update_stage_result dict→model 数据损坏

- **问题**: `InMemoryTaskRepository.update_stage_result()` 使用 `setattr(task, stage, data)` 将 Pydantic 模型字段设置为原始 dict。后续 `task.environment_check.model_dump()` 会 `AttributeError`
- **修复**: 添加 model_map 字典，根据 stage 名称自动重建正确的 Pydantic 模型
- **文件**: `app/infrastructure/persistence/task_repo.py:50-58`

### Bug B: TaskOrchestrator._execute() 幻影 RunRecord

- **问题**: `TaskOrchestrator._execute()` 传入 `run_id=f"run_{task.id}"`，但 ExecutionEngine.execute_run() 立即执行 `self._run_repo.get_by_id(run_id)` — 该 RunRecord 不存在，导致 pipeline 在第 4 阶段崩溃
- **修复**: 在调用 execute_run 之前，先通过 RunRepository.create() 创建 RunRecord
- **文件**: `app/engine/task_orchestrator.py:148-159`

### Bug C: start_task 不执行任何操作

- **问题**: `start_task` API 端点只更新状态为 "prechecking" 就返回，从不调用 `TaskOrchestrator.run_pipeline()`
- **修复**: 添加 `asyncio.create_task(_run())` 在后台启动完整八阶段 pipeline
- **文件**: `app/api/v1/tasks.py:73-91`

### Bug D: get_task_defects 返回空数据

- **问题**: `GET /tasks/{id}/defects` 只返回 `defect_count` 和 `high_risk_count`，不返回实际缺陷列表。且这两个计数储存在 TestTask 对象上，不会被赋值
- **修复**: 通过 task.run_id 查询 DefectRepository，返回完整缺陷数据
- **文件**: `app/api/v1/tasks.py:133-148`
